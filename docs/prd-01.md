微信 CLI 消息工作台需求文档

1. 产品定位

本项目是一款运行在命令行中的微信消息工作台，基于开源 Web 微信协议封装实现，用于在终端内监听、查看和回复微信消息。

它不是传统 IM 客户端，也不是模拟手机微信界面，而是面向开发者和重度命令行用户的 非全屏流式 CLI 工具。

核心体验类似 AI Coding CLI：

用户启动程序后，终端持续显示微信连接状态和新消息事件；用户可以通过命令切换联系人或群聊，进入聊天上下文后直接阅读和回复消息。

⸻

2. 产品目标

2.1 核心目标

实现一个可以长期挂在终端里的微信消息工作台，支持：

* 扫码登录微信 Web 协议
* 保持在线状态并持续监听新消息
* 以流式日志形式输出新消息
* 查看未读会话和最近会话
* 切换到指定联系人或群聊
* 进入聊天窗口后展示该会话的本地消息上下文
* 在当前聊天窗口内直接发送文本消息
* 返回全局 Inbox 监听模式
* 本地持久化消息、联系人和会话数据

2.2 设计目标

产品气质应接近：

* Codex CLI
* Claude Code
* 终端任务工作台
* 消息事件流
* 命令驱动式工作流

而不是：

* 终端版手机微信
* 聊天气泡模拟器
* 传统 IM 联系人面板
* 全屏 TUI 聊天软件

⸻

3. 需求范围

3.1 MVP 范围

第一版只实现最小可用闭环。

登录与连接

支持：

* 启动 CLI 程序
* 显示登录二维码
* 使用手机微信扫码登录
* 登录成功后进入 Inbox Monitor 模式
* 显示当前连接状态
* 断线后提示用户重新连接
* 支持保存登录会话数据，减少重复扫码

消息监听

支持：

* 持续监听新消息
* 收到新消息后在终端中流式输出
* 区分私聊和群聊消息
* 区分自己发送的消息和他人发送的消息
* 显示消息时间、会话名称、发送人、文本内容
* 当用户处于某个聊天窗口时，其他会话的新消息以后台通知形式展示

会话管理

支持：

* 查看最近会话
* 查看未读会话
* 根据联系人名称、备注名、群名打开会话
* 根据会话序号打开会话
* 进入会话后展示该会话的最近本地消息
* 返回 Inbox Monitor 模式

文本消息发送

支持：

* 在聊天窗口中输入普通文本即发送给当前会话
* 发送成功后在当前窗口显示自己发送的消息
* 发送失败时显示错误提示
* 支持显式发送命令

本地消息存储

支持：

* 保存收到的消息
* 保存自己发送的消息
* 保存联系人信息
* 保存会话信息
* 支持本地搜索消息
* 支持加载最近若干条本地历史消息

⸻

3.2 非 MVP 范围

第一版不做：

* 朋友圈
* 红包
* 转账
* 小程序卡片完整支持
* 公众号完整管理
* 多账号登录
* 消息漫游
* 拉取微信服务器完整历史记录
* 自动回复机器人
* 群管理功能
* 富文本编辑器
* 全屏 TUI 界面
* 移动端适配
* 商业化稳定协议承诺

⸻

4. 核心交互模型

系统包含两个主要运行模式：

Inbox Monitor
Chat View

4.1 Inbox Monitor

默认模式。

用户启动程序并登录后进入该模式。该模式持续监听所有新消息，并将消息作为事件流打印到终端。

示意：

● WeChat TUI
● connected as Zhibin
● watching new messages...
[14:21] 老板
  下午三点有空吗？
[14:22] 项目A群 / 王五
  接口字段改了，我发一下文档。
> 

用户可以在该模式下输入命令：

/chats
/unread
/open 老板
/search 关键词
/status
/quit

⸻

4.2 Chat View

用户打开某个联系人或群聊后进入该模式。

进入后立即输出该会话的本地消息上下文。

示意：

────────────────────────────────────────
chat: 老板 | unread 1 | local messages 24
────────────────────────────────────────
[09:12] 老板
  今天那个活动页进度怎么样？
[09:14] You
  主视觉和适配方案已经确定了，今天可以把首屏做出来。
[14:21] 老板
  下午三点有空吗？
老板 > 

在 Chat View 中：

* 普通文本默认发送给当前会话
* 斜杠命令仍然作为控制命令
* 其他会话的新消息以后台通知形式显示
* 用户可以通过 /back 回到 Inbox Monitor

⸻

5. 基础命令设计

全局命令

/chats              查看最近会话
/unread             查看未读会话
/open <name|index>  打开联系人或群聊
/contacts           查看联系人
/groups             查看群聊
/search <keyword>   搜索本地消息
/status             查看连接状态
/reconnect          重新连接
/logout             退出微信登录
/help               查看帮助
/quit               退出程序

聊天内命令

/back               返回 Inbox Monitor
/load <n>           加载更多本地历史消息
/mark read          标记当前会话已读
/search <keyword>   搜索当前会话消息
/send <text>        显式发送文本消息
/status             查看连接状态
/help               查看帮助

6. 状态展示需求

CLI 应始终让用户知道当前系统状态。

需要展示的状态包括：

* 当前账号
* 当前连接状态
* 当前模式
* 当前会话
* 未读消息数
* 最近同步时间
* 是否正在重连
* 是否登录失效
* 是否有后台新消息

状态类型包括：

init
waiting_scan
waiting_confirm
online
syncing
idle
reconnecting
offline
logout
error

8. 大致架构

建议采用分层架构，避免 CLI 直接绑定具体协议实现。

Protocol Layer
  ↓
Adapter Layer
  ↓
Runtime Layer
  ↓
Store Layer
  ↓
CLI UI Layer

⸻

8.1 Protocol Layer

负责接入底层微信 Web 协议。

第一版可以基于：

wechat4u

它负责：

* 扫码登录
* Web 微信会话维护
* 联系人同步
* 新消息监听
* 文本消息发送
* 登出和异常事件处理

该层不直接暴露给 UI。

⸻

8.2 Adapter Layer

负责屏蔽 wechat4u 的原始数据结构。

主要职责：

* 将协议事件转换成统一事件
* 将原始联系人转换成统一联系人模型
* 将原始消息转换成统一消息模型
* 封装发送消息接口
* 封装联系人搜索接口
* 处理 Web 微信 UserName 每次登录变化的问题

这一层非常重要，因为后续如果替换协议实现，只需要替换 Adapter，不影响上层 CLI 交互。

⸻

8.3 Runtime Layer

负责程序运行状态和业务流程。

主要职责：

* 维护当前模式：Inbox Monitor / Chat View
* 维护当前活跃会话
* 接收协议层消息事件
* 判断消息应该如何展示
* 路由用户输入
* 执行命令
* 处理登录、断线、重连、退出流程

核心状态包括：

currentMode
activeConversation
connectionState
unreadMap
recentConversations

⸻

8.4 Store Layer

负责本地持久化。

建议第一版使用 SQLite。

主要存储：

* 登录会话数据
* 联系人
* 群聊
* 会话
* 消息
* 附件元数据

需要注意：

wechat4u 不应被视为完整历史消息来源。CLI 中的历史记录主要来自本地数据库中已经接收和保存过的消息。

因此 /load 和 /search 面向的是本地消息库，而不是微信服务器完整历史。

⸻

8.5 CLI UI Layer

负责终端交互和输出。

主要职责：

* 渲染启动信息
* 渲染二维码提示
* 渲染状态信息
* 渲染新消息事件
* 渲染会话列表
* 渲染聊天上下文
* 处理命令输入
* 处理当前聊天输入
* 格式化错误提示

第一版建议使用普通流式 CLI，而不是全屏 TUI 框架。

⸻

10. 关键限制与风险

10.1 Web 微信协议风险

基于 Web 微信协议的客户端存在天然限制：

* 不是官方开放能力
* 部分账号可能无法登录
* 登录稳定性不可保证
* 可能受微信风控影响
* 不适合承诺商业级稳定性

产品定位应明确为：

个人工具 / 技术原型 / 本地 CLI 工作台。

⸻

10.2 历史消息限制

不能假设可以从微信服务器拉取完整历史消息。

正确设计是：

* 登录后收到的消息保存到本地
* 用户发送的消息保存到本地
* 打开会话时展示本地已有历史
* 没有本地历史时显示空状态
* 不承诺完整聊天记录同步

⸻

10.3 联系人 ID 不稳定

Web 微信中的联系人标识可能随登录会话变化。

因此：

* 不应把协议层 UserName 当永久 ID
* 每次登录后需要重新解析联系人
* 用户输入 /open 老板 时，应基于当前联系人列表搜索
* 本地收藏会话需要做弱绑定和重新匹配

⸻

11. MVP 验收标准

第一版完成后，应满足以下使用流程：

1. 用户执行 wechat-tui
2. 终端显示二维码
3. 用户扫码登录
4. CLI 显示登录成功和连接状态
5. CLI 进入 Inbox Monitor
6. 新微信消息可以实时流式输出
7. 用户输入 /chats 可以查看最近会话
8. 用户输入 /open 老板 可以进入聊天窗口
9. 进入聊天窗口后显示本地消息上下文
10. 用户输入普通文本可以发送消息
11. 发送成功后消息显示在当前窗口
12. 其他会话的新消息不会打断当前聊天，只显示后台提示
13. 用户输入 /back 返回 Inbox Monitor
14. 用户退出后，本地消息记录保留
15. 下次启动可以加载本地联系人、会话和消息数据

⸻

12. 一句话总结

这是一款基于 Web 微信协议的 非全屏流式微信 CLI 工作台：它通过事件流监听新消息，通过命令切换会话，通过本地缓存提供聊天上下文，让微信消息处理变成一种终端里的工作流。
